# Seed Data Implementation Plan

## Steps

1. Create `scripts/seed_data.py`.
2. Add a small internal data model describing users, circles, tags, teams, memberships, and invitations for `demo` and `stress`.
3. Implement dataset detection helpers based on user and circle prefixes.
4. Implement safe reset helpers that delete only records belonging to the selected dataset.
5. Implement deterministic seed creation that rebuilds the selected dataset after cleanup.
6. Add automated tests for dataset creation, repeated execution, and reset safety.

## Script Responsibilities

- Parse CLI args: dataset name plus optional `--reset`.
- Ensure tables exist before mutation.
- Open a session against the existing SQLite database.
- If `--reset` is provided, clean the selected dataset only.
- Otherwise, rebuild the selected dataset deterministically.
- Print a short summary of created or deleted counts.

## Test Plan

Add a dedicated test module for the script helpers rather than shelling out to a subprocess.

Required coverage:

- `demo` seed creates expected counts and dataset markers.
- `stress` seed creates a richer set of circles, teams, invitations, and tags.
- Re-running `demo` does not accumulate duplicates.
- `demo --reset` preserves unrelated users, circles, teams, and invitations.
- `stress --reset` preserves unrelated data and does not remove `demo` seed data.

## Acceptance Criteria

- The script can be run manually against the local SQLite database.
- Seeded users use hashed passwords from the real user service.
- Reset logic is dataset-scoped and relationship-safe.
- Tests prove seeded cleanup does not delete original data.
